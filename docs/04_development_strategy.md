# 📘 Development Strategy Document (균형형 전략)

## 1. 문서 목적

본 문서는 자동매매 시스템 개발 시 **일관성, 안정성, 확장성**을 유지하기 위한 개발 규칙을 정의한다.

특히 본 프로젝트는:

* 실제 자금이 투입되는 시스템
* 상태 머신 + 이벤트 기반 구조
* 장부(계좌/포지션) 정합성이 핵심

이므로 일반 프로젝트보다 **엄격한 규칙이 필요하다.**

---

# 2. 최우선 원칙 (절대 규칙)

## 2.1 설계 문서 우선 원칙

다음 문서는 **항상 기준으로 삼는다**:

* 요건정의서
* 기본설계서
* 상세설계서

👉 특히 다음 파일은 매 작업을 진행할 시에 확인해야 한다:

* 01_requirements_specification.md 
* 02_basic_design_specification.md 
* 03_detailed_design_specification.md 

### 규칙

* docs와 다른 구현은 금지
* 변경이 필요하면 → **문서 수정 → 구현 수정 순서**

---

## 2.2 상태/장부 무결성 원칙

다음은 절대 위반 금지:

* `OrderEvent` 없이 포지션 변경 금지
* 상태 머신 우회 금지
* 현금/포지션 직접 수정 금지

👉 이유:

* 이 구조가 깨지면 실제 손실 발생

---

## 2.3 책임 분리 원칙

다음 규칙을 반드시 지킨다:

* Strategy → 주문 실행 금지
* Execution → 전략 판단 금지
* Portfolio → 주문 실행 금지
* Infrastructure → 비즈니스 판단 금지

---

# 3. Git / 버전 관리 전략

## 3.1 브랜치 전략

```text
main        → 안정 버전 (항상 실행 가능)
develop     → 개발 통합 브랜치
feature/*   → 기능 개발
fix/*       → 버그 수정
```

### 규칙

* main 직접 커밋 금지
* 모든 작업은 feature 브랜치에서 시작
* 완료 후 develop으로 병합

---

## 3.2 커밋 규칙

### 형식

```text
[type] 내용 (주 언어는 한국어로 작성)

예:
[feat] breakout strategy 초기 구현
[fix] 주문 상태 전환 버그 수정
[refactor] 포트폴리오 갱신 구조 개선
[test] 주문 상태머신의 테스트 추가
```

### 타입 목록

* feat
* fix
* refactor
* test
* docs

---

## 3.3 커밋 원칙

* 한 커밋 = 하나의 의미
* 상태 머신/장부 변경은 반드시 별도 커밋
* 깨진 상태로 커밋 금지

---

# 4. 개발 규칙

## 4.1 구현 순서 규칙

반드시 아래 순서를 따른다:

1. 상세설계 확인
2. 인터페이스 정의
3. 구현
4. 테스트
5. 로그 확인

---

## 4.2 금지 사항

❌ Strategy에서 Broker 호출
❌ Position 직접 수정
❌ 상태 머신 우회
❌ 하드코딩 파라미터
❌ 테스트 없이 핵심 로직 변경

---

## 4.3 필수 로그

다음은 반드시 로그 남긴다:

* 주문 요청
* 주문 상태 변화
* 체결 이벤트
* 포지션 변경
* 손익 계산

---

# 5. 테스트 전략

## 5.1 반드시 테스트해야 하는 영역

### 1) 상태 머신

* 정상 전이
* 금지 전이
* 종료 상태 처리

### 2) 장부

* 매수/매도 체결
* 부분 체결
* 취소
* 비용 반영

### 3) 전략 흐름

* 종가 확정 → 다음날 시가 실행

---

## 5.2 테스트 규칙

* 상태 머신 변경 → 테스트 필수
* 장부 로직 변경 → 테스트 필수
* 테스트 없는 변경 → 금지
